# 객체 지향(Object Oriented Programming)이란?




객체 지향 프로그래밍의 3가지 특성을 흔히 "캡슐화", "상속성", "다형성"이라고 한다. 각각의 특징을 간단히 요약하면


    1. 캡슐화 (Encapsulation)
	  - 객체의 멤버 데이터를 private 영역에 놓고, public 멤버 함수를 통해서 접근하는 것.
	  - 실제로 처리되는 데이터 부분을 외부에 드러나지 않도록 하여 정보를 은닉하고 보호할 수 있음.
	  - 데이터와 기능을 하나로 묶어 관리하므로 각각의 객체는 독립적인 역할을 할 수 있음.
	  
	2. 상속성 (Inheritance)
	  - 하나의 클래스가 가진 특징(멤버 데이터 및 멤버 함수)을 다른 클래스가 그대로 물려받을 수 있는 것.
	  - 이미 작성된 클래스를 활용하여 새로운 특징을 추가한 클래스를 만들 수 있어 기존 코드의 재활용성이 높아짐.
	
	3. 다형성 (Polymorphism)
	  - 약간 다른 방식으로 동작하는 함수를 동일한 함수 명으로 호출하는 것.
	    또는, 동일한 명령의 해석을 연결된 객체 또는 데이터에 의존하는 것
	  - ex. 오버라이딩, 오버로딩

	4. 추상화 (Abstraction)
	  - 객체들의 공통적인 특징 (데이터 또는 함수)를 도출하는 것.
	  - 도출된 공통적인 특징을 바탕으로 클래스를 정의하고, 클래스를 바탕으로 세부 객체를 만들어 나감.




요즘 C++ 문법 공부를 하며 객체 지향에 대해서 공부하고 있는데, 이 개념이 정말 재미있다. 실제로 사람이 사고하는 방식을 본따 만들었다고 들은 것 같기도 한데, (그래서 절차 지향 프로그래밍에 비해 좀 더 직관적인 프로그래밍이 가능하다고 한다) 어떤 객체의 속성들을 정의하고(멤버 변수) 그 객체가 할 수 있는 행동(멤버 함수)를 정의해 묶어서 관리하겠다는 것이다. 그리고 그렇게 객체를 정의한 덕에, 비슷한 속성을 가진 다양한 인스턴스들 또는 새로운 객체를 만들어 낼 수 있어 코드의 재사용성과 가독성 등이 높아지고 결국 생산성이 향상된다고 한다.



예제 코드를 짜며 공부를 좀 하다보니 예전에 대학생때 공부했던 내용들이 떠올랐다. 차량 동역학 및 에너지 제어 연구실에서 학부 연구생 활동을 하며 매트랩/시뮬링크로 차량 동역학 모델을 (엔진부터 바퀴까지) 만들고, 외부 조건들을 변화시켜가며 차량의 거동을 시뮬레이션하곤 했었는데, 그때 했던 것이 딱 객체 지향의 캡슐화와 추상화 내용이었던 것이다.




# 차량 동역학 시뮬링크 모델로 보는 객체 지향 


![Alt text](/assets/img/posts/simulink_1.png)




위 캡쳐는 도로의 경사도가 차량 주행 성능에 영향을 미치는 것을 시뮬레이션하기 위한 시뮬링크 모델이다. 제어기 모델(Controller)과 차량 모델(My Car), 두가지 객체가 정의되어 있다. 각 객체는 서로 상대 객체가 어떻게 작동하는지 알 필요는 없으며, 제어기는 목표 차속과 현재 차속 그리고 도로의 경사도만 입력으로 받아 차량 속도 제어에 필요한 악셀 페달과 브레이크 페달의 값만 계산한다. 차량 모델은 제어기가 어떤 원리로 작동하는지 알 필요는 없으며 그냥 차량 동역학 모델에 맞게 현재 차량의 속도만 계산해서 출력하면 되는 것이다.



![Alt text](/assets/img/posts/simulink_2.png)



차량 시스템을 열어보면 위 캡쳐와 같은데, 차량 동역학적인 부분과 엔진 부분이 나뉜다. 차량 동역학 부분은 중력, 타이어와 바닥 사이의 마찰력, 엔진에서 부터 전달받은 구동 토크 등을 종합하여 최종적으로 F=ma 를 이용해 가속도를 구하고 속력을 계산한다. 엔진은 현재 RPM과 악셀 페달 수치, 차속을 입력으로 받아 출력 토크와 출력 RPM을 계산한다. 

이때에도 엔진 부분과 차량 부분 사이에는 각자 필요한 정보만 정의된 인터페이스 내에서 주고 받으면 되고, 세부적으로 어떻게 작동할지는 전혀 상관이 없다.



![Alt text](/assets/img/posts/simulink_3.png)



엔진 시스템을 다시 또 열어서 자세히 살펴보면, 엔진과 토크컨버터 모델, AT 기어 변속 로직, 트랜스미션 모델 등으로 나눌 수 있다. 여기서도 마찬가지로 각 모델들이 어떻게 입력값을 출력값으로 계산하는지는 상관없으며 서로 정의된 인터페이스 내에서 각각 역할만 다해주면 된다.




그리고 이렇게 정의된 시스템과 그 안의 세부적인 서브 시스템들은 내부 파라미터만 변경하여 비슷하지만 다른 모델로 변경할 수 있다. 똑같은 자동차이지만 엔진 출력과 제동 성능을 높여 스포츠가를 만들 수도 있고, 반응성은 조금 떨어질지라도 엔진의 최적 구동 상태를 가능한 유지하도록 해 연비가 높은 자동차를 만들 수도 있는 것이다. 한번만 모델을 잘 설계해두면 그 뒤로는 다양하게 자동차의 거동을 시뮬레이션 할 수 있게 된다.




차량 모델을 처음부터 한번 짜는게 어려웠지 한번 만든 이후로는 차종별로 무게중심과 엔진맵, 타이어 마찰 계수 같은 것을 바꿔가며 시뮬레이션을 엄청 돌려봤던 기억이 난다. 저 시절에는 객체 지향이라는 개념을 전혀 몰랐지만 사실 이게 바로 객체 지향이었구나 하는 생각을 요즘 하고 있다. (지금 돌이켜 생각해보니 교수님이 이게 객체 지향이다라는 말씀을 하셨던 것 같기도 하고..)


    1. 캡슐화 (Encapsulation)
	  - 엔진이나 차량 무게 중심 등 차량의 디테일한 파라미터들은 외부에 드러나지 않지만, 시뮬레이션 시 차량 모델을 사용할 수 있음.
	  
	2. 상속성 (Inheritance)
	  - 한번 구현된 차량 모델을 그대로 복사해서 일부 파라미터를 수정하거나 다른 기능을 덧붙여 새로운 차량을 만들 수 있음.
	
	3. 다형성 (Polymorphism)
	  - 정확한 비유는 아닌 것 같지만, 일부 기능은 입력값이나 기존 상태에 따라 기능이 일부 바뀌는 경우도 있음.

	4. 추상화 (Abstraction)
	  - 각 모듈 사이에 인터페이스만 명확히 정의되어 있으면, 서로 상대의 상세한 구현 내용은 몰라도 상관없으며, 그러한 기능들을 묶어 더 큰 기능을 만들 수 있음.




이렇게 예전에 배웠던 개념에 빗대어 객체 지향을 이해하려고 하니 더 흥미롭기도 하고 이해도 잘 되었다. 



# 객체 지향 프로그래밍의 장점? 단점?



객체 지향 프로그래밍을 통해 소프트웨어의 생산성 향상, 실세계에 대한 쉬운 모델링 등이 가능하다고 한다. 그리고 경우에 따라서는 보안성이 높아질 수도 있다고 한다. 차량 모델을 생각해보면 다 맞는 말이다. 대신 메모리 공간이나 실행 속도에 대해서는 좀 더 손해를 볼 수도 있다고 한다. 임베디드 제어기에 들어가는 SW 도 실시간성 때문에 객체 지향은 포기하고 C언어로 하드 코딩 경우도 많은 것을 보면 알 수 있다.
